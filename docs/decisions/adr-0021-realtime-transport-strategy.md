# ADR-0021: 실시간 클라이언트 전송 전략 — 채팅 WebSocket / 알림 SSE

## Status
Accepted

---

## Context

Routinely는 두 가지 실시간 기능을 제공한다:

1. **채팅(Chat)** — 그룹 채팅방에서 다수의 사용자가 메시지를 주고받는다.
2. **알림(Notification)** — 서버가 루틴 시작 알림, 챌린지 이벤트 알림 등을 클라이언트에 전달한다.

두 기능 모두 서버-클라이언트 간 실시간 데이터 전달이 필요하나,
데이터 흐름 방향이 근본적으로 다르다.

| 기능 | 데이터 흐름 |
|------|-------------|
| 채팅 | 클라이언트 → 서버 / 서버 → 클라이언트 (양방향) |
| 알림 | 서버 → 클라이언트 (단방향) |

사용 가능한 실시간 전송 기술:

- **WebSocket** — 단일 TCP 연결 기반 양방향 전이중 통신
- **SSE (Server-Sent Events)** — HTTP 기반 서버→클라이언트 단방향 스트리밍
- **Long Polling** — HTTP 반복 요청으로 실시간 흉내
- **FCM (Firebase Cloud Messaging)** — 모바일/웹 푸시 알림 서비스

---

## Decision

- **채팅**: **WebSocket (STOMP over WebSocket)** 사용
- **알림**: **SSE (Server-Sent Events)** 사용

---

## Options Considered

### 채팅 전송 기술

#### Option 1: WebSocket (채택)

클라이언트와 서버가 단일 TCP 연결을 유지하며 양방향으로 자유롭게 메시지를 교환한다.

**채택 이유:**

- 채팅은 클라이언트가 메시지를 **보내는 동시에** 다른 사람의 메시지를 **받아야 한다** → 양방향 필수
- STOMP 프로토콜을 사용하면 `/app/`, `/topic/`, `/queue/` 기반의 구독 모델로 채팅방 단위 라우팅이 가능하다
- 연결을 유지하므로 메시지 수신 지연이 낮다

#### Option 2: HTTP Polling (기각)

| 문제 | 설명 |
|------|------|
| 불필요한 요청 | 새 메시지가 없어도 주기적으로 요청 발생 |
| 지연 | 폴링 간격만큼 메시지 수신 지연 |
| 서버 부하 | 사용자 수 × 폴링 빈도만큼 요청 증가 |

---

### 알림 전송 기술

#### Option 1: SSE (채택)

HTTP 응답을 끊지 않고 서버가 이벤트를 스트리밍으로 밀어주는 방식.

**채택 이유:**

- 알림은 **서버 → 클라이언트 단방향** → WebSocket의 양방향 기능이 불필요하다
- 표준 HTTP 위에서 동작하여 프록시·방화벽 호환성이 높다
- 브라우저 내장 `EventSource` API가 자동 재연결을 처리한다
- WebSocket 대비 구현·운영 부담이 낮다

#### Option 2: WebSocket (기각)

| 문제 | 설명 |
|------|------|
| 과잉 설계 | 알림은 단방향인데 양방향 채널 유지는 불필요한 오버헤드 |
| 별도 인스턴스 관리 | 채팅 WebSocket 서버와 알림 WebSocket 서버를 분리해야 하거나 책임이 혼재됨 |

#### Option 3: FCM (기각)

| 문제 | 설명 |
|------|------|
| 운영 복잡도 | `fcm_tokens` 관리, 토큰 갱신, 디바이스별 처리 필요 |
| MVP 오버킬 | 웹 전용 MVP 단계에서 모바일 푸시 인프라 도입은 비용 대비 이득 없음 |
| 탭 off 알림 불필요 | MVP 요구사항에 탭 off 상태 알림 없음 |

## 기술 비교 요약

| 항목 | WebSocket | SSE | Long Polling |
|------|:---------:|:---:|:------------:|
| 방향 | 양방향 | 단방향 (서버→클라이언트) | 단방향 (흉내) |
| 프로토콜 | WS / WSS | HTTP | HTTP |
| 자동 재연결 | 직접 구현 필요 | 브라우저 기본 제공 | 매 요청 재연결 |
| 구현 복잡도 | 높음 (STOMP 등 필요) | 낮음 | 낮음 |
| 서버 부하 | 낮음 (연결 유지) | 낮음 (연결 유지) | 높음 (반복 요청) |
| 적합 용도 | 채팅, 게임, 협업 도구 | 알림, 피드, 진행 상태 | 레거시 호환 |

---

## 결정 원칙

> **양방향이 필요한가?**
> - Yes → WebSocket
> - No (서버 → 클라이언트만) → SSE

채팅은 클라이언트가 메시지를 **전송**해야 하므로 WebSocket이 필수다.
알림은 서버가 클라이언트에게 **통보**만 하면 되므로 SSE로 충분하다.

---

## 멀티 인스턴스 확장

SSE와 WebSocket 모두 인스턴스별로 클라이언트 연결을 관리한다.
스케일 아웃 시 브로드캐스트 문제가 발생하므로 각 서비스에서 다음 전략을 사용한다:

| 서비스 | 확장 전략 |
|--------|-----------|
| Chat (WebSocket) | Kafka 기반 멀티 인스턴스 브로드캐스트 (ADR-0016 참조) |
| Notification (SSE) | 사용자별 연결 라우팅 — 동일 인스턴스 연결 보장 (Sticky Session 또는 연결 레지스트리) |

## Consequences

### Positive

- 기능별 통신 특성에 맞는 프로토콜 선택 → 불필요한 오버헤드 제거
- SSE는 HTTP 기반이므로 Gateway 통과 및 CORS 처리가 단순하다
- WebSocket(STOMP)은 채팅방 구독 모델을 명확하게 표현한다
- FCM 미도입으로 MVP 운영 복잡도 유지

### Negative

- SSE는 탭을 닫은 상태에서 알림 수신 불가
  - 완화: 재접속 시 미수신 알림 조회 API 제공 / 향후 Web Push + Service Worker로 확장
- WebSocket은 일부 구형 프록시에서 연결이 끊길 수 있음
  - 완화: WSS(TLS) 사용 시 대부분 해결됨

---

## Future Considerations

- 탭 off 상태 알림이 필요해지면 Web Push + Service Worker 확장
- Notification SSE 멀티 인스턴스 확장 시 연결 레지스트리(Redis) 도입 검토
