# ADR-0002: MSA 구조 유지 + Docker Compose 기반 배포 전략

- **Status**: Accepted
- **Date**: 2026-02-15
- **Author**: Routinely Project

---

## 1. Context

Routinely는 마이크로서비스 아키텍처(MSA) 기반으로 설계된다.

예상 서비스 구성:
- API Gateway
- Routine/Challenge Service
- Chat Service
- Notification Service
- (확장) Report / 기타 서비스

하지만 본 프로젝트의 목적은:

- 상용 서비스 운영이 아닌 **포트폴리오 및 학습 목적**
- 지인 대상 가벼운 공유
- **비용 최소화**
- 빠른 개발 및 반복 배포

따라서 다음과 같은 선택지가 존재했다:

1. Kubernetes(EKS/AKS) 기반 배포
2. 단일 VM에서 수동 실행
3. Docker Compose 기반 통합 실행

---

## 2. Decision

**MSA 구조는 유지하되, 배포는 Docker Compose 기반으로 진행한다.**

### 구체적 선택

- 각 마이크로서비스는 개별 Docker 이미지로 빌드
- Docker Compose를 통해:
    - 서비스 컨테이너
    - PostgreSQL
    - Redis
    - (선택) Kafka
      등을 한 번에 실행
- 단일 VM 또는 저비용 클라우드 인스턴스에 배포
- Kubernetes(EKS/AKS)는 도입하지 않는다

---

## 3. Rationale (결정 이유)

### 3.1 프로젝트 목표와의 정합성

Routinely는:

- 대규모 트래픽을 감당하는 상용 서비스가 아님
- 지인 대상 테스트 및 학습 목적
- 고가용성/오토스케일링이 필수 요구사항이 아님

Kubernetes는 현재 요구사항 대비 과도한 복잡성을 초래한다.

---

### 3.2 복잡도 대비 효용

Kubernetes 도입 시 추가로 필요한 작업:

- 클러스터 구성
- Ingress 설정
- Helm 또는 매니페스트 관리
- 리소스 요청/제한 관리
- 네트워크 정책 이해
- CI/CD 파이프라인 확장

현재 프로젝트 규모와 목적을 고려할 때 학습 비용이 과도하다고 판단했다.

반면 Docker Compose는:

- 서비스 간 네트워크 자동 구성
- 환경 변수 관리 용이
- 로컬 환경과 배포 환경의 유사성 유지
- 빠른 실행 및 종료 가능

---

### 3.3 MSA 학습 목적은 유지 가능

Docker Compose를 사용하더라도:

- 서비스는 여전히 독립 프로세스로 실행
- 데이터베이스 분리 유지
- 서비스 간 gRPC/이벤트 통신 유지
- 장애 격리 구조 유지

즉, **아키텍처적 학습 가치는 유지된다.**

---

## 4. Alternatives Considered

### 4.1 Kubernetes (EKS / AKS)

**장점**

- 오토스케일링
- 롤링 배포
- 고가용성 구성 가능
- 실무 어필 가능성

**단점**

- 운영 복잡도 증가
- 비용 상승
- 학습 곡선 가파름
- 프로젝트 목적 대비 과도함

→ 현 단계에서는 오버엔지니어링으로 판단

---

### 4.2 단일 VM 수동 실행 (Docker 없이 jar 실행)

**장점**

- 가장 단순한 구조
- Docker 없이도 실행 가능

**단점**

- 환경 재현성 낮음
- 의존성 관리 어려움
- 서비스 확장 시 관리 어려움

→ 컨테이너 기반 실행이 더 적합하다고 판단

---

## 5. Consequences

### 긍정적 영향

- 빠른 개발 및 배포 가능
- 비용 최소화
- 로컬/배포 환경 일관성 유지
- 운영 복잡도 감소
- MSA 구조 학습 유지

### 부정적 영향

- 오토스케일링 미지원
- 고가용성 구성 없음
- 롤링 업데이트 자동화 미지원
- 프로덕션 수준의 운영 경험은 제한적

---

## 6. Future Considerations

다음 조건이 충족될 경우 Kubernetes 도입을 재검토할 수 있다:

- 실제 사용자 트래픽 증가
- 다중 인스턴스 운영 필요
- 고가용성 요구 증가
- 운영 자동화 요구 증가

그 전까지는 Docker Compose 기반 운영 전략을 유지한다.

---

## 7. Summary

Routinely는 **MSA 구조를 유지하면서도 운영 복잡성을 최소화하기 위해 Docker Compose 기반 배포 전략을 채택한다.**

이 결정은 프로젝트의 목적(학습/포트폴리오/저비용 운영)에 가장 적합한 현실적인 선택이다.
