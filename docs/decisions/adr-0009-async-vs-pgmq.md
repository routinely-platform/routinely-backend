# ADR-0009: @Async 기반 비동기 vs PGMQ 기반 비동기 처리 전략

## Status
Accepted

---

## Context

Routinely는 다음과 같은 비동기 작업을 필요로 한다:

- 루틴 완료 후 후처리 작업
- 알림 예약(sendAt 기반 due 처리)
- 외부 알림 발송 (SSE 등)
- 재시도 로직이 필요한 작업

Spring에서는 `@Async`를 통해 간단히 비동기 처리를 구현할 수 있다.
또 다른 선택지는 PostgreSQL 기반 큐(PGMQ)를 사용하는 것이다.

비동기 처리 전략을 명확히 정의할 필요가 있었다.

---

## Option 1: @Async 기반 비동기 처리

### 개념

Spring의 `@Async`를 사용하여
메서드를 별도의 스레드에서 실행한다.

### 예시

```java
@Async
public void sendNotification(NotificationPayload payload) {
    externalApi.send(payload);
}
```
### 장점

- 구현이 매우 단순함
- 추가 인프라 필요 없음
- 빠르게 적용 가능

### 단점

- 애플리케이션 재시작 시 작업 유실 가능
- 작업 상태 추적 어려움
- 예약(due) 처리에 부적합
- 재시도 로직 직접 구현 필요
- 분산 환경 확장 어려움

---

## Option 2: PGMQ 기반 비동기 처리

### 개념

PostgreSQL 기반 큐에 작업을 저장하고  
워커가 해당 작업을 처리한다.

### 처리 흐름

1. 비즈니스 트랜잭션 커밋
2. PGMQ에 작업 enqueue
3. 워커가 큐를 polling
4. 작업 처리 성공 시 ack/delete
5. 실패 시 재시도

### 장점

- 작업 영속성 보장 (Durability)
- 재시도 정책 구현 용이
- 예약(due) 기반 처리 가능
- 작업 상태 추적 가능
- 워커 수평 확장 가능

### 단점

- 구현 복잡도 증가
- PostgreSQL 부하 증가
- 워커 운영 관리 필요

---

## Decision

Routinely는 **서비스 내부 비동기 작업 처리에 대해  
PGMQ 기반 비동기 방식을 채택한다.**

`@Async`는 단순 로컬 비동기 처리(로그, 비핵심 작업)에만 제한적으로 사용한다.

---

## Rationale

### 핵심: 이벤트(Event)와 작업(Action)은 다르다

Kafka Consumer가 이벤트를 수신한 뒤 수행하는 일은 더 이상 "이벤트 소비"가 아니라 **"작업 실행"**이다.

```
[Kafka 이벤트]                        [서비스 내부 작업]
routine.execution.done                 → 통계 집계
  (사실: 루틴이 완료됨)                  → 알림 발송
                                       → 랭킹 업데이트
```

- **이벤트(Event)** — "루틴이 완료되었다"는 사실(fact). 브로드캐스트의 대상.
- **작업(Action)** — "알림을 보내라", "통계를 집계하라"는 행위. 실행과 완료의 대상.

이벤트 소비 실패 시 재처리, 분기 처리, 예약 실행은
이벤트를 다시 소비하는 것이 아니라 **작업을 수행하는 것**이다.

따라서 이 작업들은 Kafka의 retry가 아니라 **작업 큐(PGMQ)에서 관리하는 것이 본질에 맞다.**

### Kafka와 PGMQ의 역할 분리

| 구분 | Kafka | PGMQ |
|---|---|---|
| 성격 | 이벤트 (사실 전달) | 작업 (행위 실행) |
| 메시지 의미 | "~가 발생했다" | "~를 수행하라" |
| 재시도 대상 | 메시지 전달 | 작업 실행 |
| 예약/지연 | 부적합 | 적합 (sendAt 기반 due 처리) |
| 분기 처리 | 부적합 | 적합 (작업 유형별 분기) |
| 상태 추적 | 없음 (offset만 존재) | 가능 (큐 항목별 상태 관리) |

### 흐름 예시: 루틴 완료 → 알림 발송

```
1. routine-service: 루틴 완료 → Outbox → Kafka publish
2. notification-service: Kafka 이벤트 수신 (이벤트 소비)
3. notification-service: PGMQ에 "알림 발송" 작업 enqueue (이벤트 → 작업 전환)
4. notification-service: Worker가 PGMQ에서 작업을 꺼내 SSE 알림 발송 (작업 실행)
5. 발송 실패 → PGMQ에서 backoff 후 재시도 (작업 재시도)
```

이벤트 수신(2단계)은 Kafka가 담당하고,
작업 실행과 재시도(3~5단계)는 PGMQ가 담당한다.

### 기타 근거

1. 알림 시스템은 예약 기반(due) 처리가 필수이다.
2. 외부 API 호출은 재시도 로직이 필요하다.
3. 작업 유실은 허용되지 않는다.
4. MSA 환경에서 워커 확장이 필요할 수 있다.
5. 작업의 내구성(Durability)을 보장해야 한다.

`@Async`는 메모리 기반 스레드 실행 방식이므로
애플리케이션 장애 발생 시 작업이 유실될 수 있다.

PGMQ는 작업을 데이터베이스에 저장하여
내구성과 안정성을 확보할 수 있다.

---

## Usage Rule

- 단순 비동기 로그 처리 → `@Async` 사용 가능
- 외부 알림 발송 → PGMQ 사용
- 예약 기반 작업 → PGMQ 사용
- 재시도 필요 작업 → PGMQ 사용
- 서비스 간 이벤트 전달 → Kafka 사용
- 즉시 응답이 필요한 요청 → gRPC 사용

---

## Consequences

### Positive

- 작업 안정성 확보
- 재시도 정책 일관성 확보
- 예약 처리 구조 명확화
- 분산 환경 대응 가능

### Negative

- 코드 복잡도 증가
- 운영 부담 증가
- PostgreSQL 리소스 사용 증가

---

## Architectural Principle

비동기는 단순한 스레드 분리가 아니라
**"내구성과 재시도를 보장하는 작업 처리 방식"**이어야 한다.

이벤트 소비 실패 시 재처리는 "이벤트를 다시 듣는 것"이 아니라
**"작업을 다시 수행하는 것"**이다.

Routinely는 이 원칙에 따라
Kafka는 이벤트 전달에, PGMQ는 작업 실행에 사용한다.
